# Frontend SEO Implementation Plan

## Context

**Stack:** React 18 + Vite + react-router-dom v6 (Client-Side SPA)  
**Current State:** Zero SEO implementation — no dynamic titles, meta descriptions, Open Graph, Twitter Cards, helmet library, or favicon. All 14 routes (including `/login`) render identical `<title>` from `index.html`.  
**Goal:** Make this court scheduling app indexable and shareable — at minimum correct title/description per page, full social-card support, and a factual baseline for crawlers.

---

## Task Breakdown

### Task 1 – Install `react-helmet-async`

- **Why:** `react-helmet-async` is the standard, concurrent-safe way to manage `<head>`
  in React SPAs. It lets each page component declare its own `<title>`, meta
  descriptions, canonical URLs, OG / Twitter tags, robots directives, etc.
- **File:** `Frontend/package.json`
- **Steps:**
  1. Add `react-helmet-async` as a dependency.
  2. No import in `index.html` — HelmetAsync mounts/unmounts via the React component
     tree so all rendered tags arrive in `<head>`.

### Task 2 – Wrap the app with `HelmetProvider`

- **File:** `Frontend/src/App.jsx`
- **Steps:**
  1. Import `HelmetProvider` from `react-helmet-async`.
  2. Wrap the component tree (inside `App()` or at the `<AuthProvider>` level)
     with `<HelmetProvider>` so all descendants can use `<Helmet>` or `useHelmet`.
  3. Leave authentication/public routes to their own `<Helmet>` declarations
     — login page needs its own title/description too.

### Task 3 – Create `src/seo/seoDefaults.js`

- **Why:** A single source of truth for app-wide defaults prevents duplication and
  ensures no `<head>` attrs are accidentally missed on any page.
- **File:** `Frontend/src/seo/seoDefaults.js`
- **Export:**
  - `SEO_PROPS_DEFAULT` object with keys:
    - `siteName` → `"Court Scheduling System"`
    - `defaultTitle` → `"Court Scheduling System"`
    - `titleTemplate` → `"%s | Court Scheduling System"`
    - `defaultDescription` → canonical site blurb
    - `robots` → `"index, follow"`
    - `googlebot` → `"index, follow"`
  - `SITE_URL` constant (e.g. `https://court-scheduling.example.com`) — used
    for canonical URLs and OG URL tag; updateable per deployment env.
  - `getMetaArray` helper: builds `[{ name, content }, …]` arrays from a props
    object so it's easy to spread into `<meta />` elements.

### Task 4 – Create `src/seo/PageSEO.jsx` (reusable component)

- **Why:** Every page uses the same set of tags. One component keeps usage
  consistent.
- **File:** `Frontend/src/seo/PageSEO.jsx`
- **Props:**
  | Prop | Type | Purpose |
  |------|------|---------|
  | `title` | `string` | Page-specific `<title>` (suffixes `titleTemplate` in Helmet) |
  | `description` | `string` | `<meta name="description">` / OG description |
  | `canonical` | `string` | Full canonical URL |
  | `robots` | `string` | Defaults to `"index, follow"` |
  | `ogImage` | `string` | Absolute URL of the OG thumbnail |
  | `ogType` | `string` | Defaults to `"website"` |
  | `structuredData` | `object` | Passed to `<Helmet>` `<script type="application/ld+json">`, Optional |
- **Renders inside `Helmet`:**
  - `<title>` / `<meta property="og:title">`
  - `<meta name="description">` / `<meta property="og:description">`
  - `<link rel="canonical">`
  - `<meta name="robots">`
  - Open Graph: `og:type`, `og:site_name`, `og:url`, `og:image`, `og:description`, `og:title`
  - Twitter Card: `twitter:card`, `twitter:image`, `twitter:site`, `twitter:title`, `twitter:description`
  - Structured data `<script type="application/ld+json">` — will be defined per-page

### Task 5 – Add structured data per page

- **Why:** Google & Bing read JSON-LD to generate rich results (breadcrumbs,
  snippets, knowledge panels). Each route maps to one piece of content — the
  structured data follows that shape.
- **Pattern:** Pass a `structuredData` object to `<PageSEO>` for routes where an
  article/commercial/organization schema applies.
- **Schemas to implement:**
  - `/cases`, `/judges`, `/courtrooms`, `/schedule`, `/calendar`, `/reports`,
    `/gap-analysis` → **Organization** schema (basic firm / court context)
  - `/ai-insights` → **WebPage** with `about` pointing to the insights content,
    optionally **FAQPage** if a page-level FAQ is available in the future
  - `/dashboard` → **WebPage** schema
  - `/login`, `/users`, `/settings`, `/profile` → no structured data
  - Every page → **BreadcrumbList** with `Home > [PageName]`

- **Why:** Dynamic `<title>` and `<meta>` per route is the minimum SEO signal
  crawlers use. Without it every page scores as a duplicate of the default `index.html`
  title with no unique description.

### Task 7 – Replace `index.html` meta block with canonical defaults

- **Why:** Helmet acts on the live DOM at runtime. `index.html` `<title>` is the
  fallback shown before JS hydrates — it should still be accurate.
- **File:** `Frontend/index.html`
- **Update the `<head>` block:**
  - Keep `lang="en"`, charset, viewport, preconnect, fonts.
  - Set `<title>` to the `titleTemplate` from `seoDefaults` (e.g. `"Court Scheduling System"`).
  - Add `<meta name="description">` matching `defaultDescription`.
  - Add OG baseline tags in the HTML shell so they are present even before JS:
    - `og:site_name`, `og:type`, `og:title`, `og:description`, `og:url`, `og:image`
  - These will be **overwritten** at runtime by Helmet tags rendered by
    `<PageSEO>` on each page.
  - Add `<link rel="canonical">` defaulting to `SITE_URL`.
  - Add Twitter Card baseline: `twitter:card=summary_large_image`.

### Task 8 – Add favicon files

- **Why:** Search results and social cards use favicons in listings; absence
  looks unprofessional and Bing/Google explicitly call out missing favicons as a
  signal issue.
- **Files:**
  - `Frontend/public/favicon.svg` — simple SVG court-balance or court-gavel icon.
  - `Frontend/public/favicon-32x32.png` — 32 px PNG fallback.
  - `Frontend/public/favicon-16x16.png` — 16 px PNG fallback.
  - `Frontend/public/apple-touch-icon.png` — 180 × 180 px PNG for iOS home-screen.
  - Confirms `index.html` references `/favicon.svg` match an actual file.

### Task 9 – Add `manifest.json`

- **Why:** Required for PWA installability and provides app metadata that browsers
  surface in "Add to Home Screen" and in search-rich snippets.
- **File:** `Frontend/public/manifest.json`
- **Contents:**
  - `name`, `short_name` → `"Court Scheduling System"`
  - `description` → same string as `seoDefaults.defaultDescription`
  - `start_url` → `"/"`
  - `display` → `"standalone"`
  - `background_color`, `theme_color`
  - `icons` → references to the PNG favicons created in Task 8
- **`index.html` update:** add `<link rel="manifest" href="/manifest.json">`.

### Task 10 – Add `robots.txt` and `sitemap.xml`

- **Why:** Tells crawlers which directories to index and surfaces the URL set for
  declarative SEO crawling tools.
- **Files:**
  - `Frontend/public/robots.txt` — `User-agent: *`, `Allow: /`, `Sitemap: https://<domain>/sitemap.xml`
  - `Frontend/public/sitemap.xml` — one `<url>` entry per defined route path with
    `<lastmod>` from the most recent build date and `<changefreq>` + `<priority>`
    tuned per route importance:
    - `/` / dashboard → `weekly`, `1.0`
    - `/cases` → `daily`, `0.9`
    - `/schedule`, `/calendar` → `daily`, `0.8`
    - `/ai-insights` → `weekly`, `0.4` (auto-generated content, less stable)
    - `/reports`, `/gap-analysis` → `weekly`, `0.7`
    - remaining pages → `monthly`, `0.5`

### Task 11 – Harden Semantic HTML in shared layout components

- **Why:** Heading hierarchy, landmarks, and ARIA roles help screen readers and
  give crawlers context about page structure — on their own they are low-weight
  SEO signals but cost zero to add alongside Tasks 1-10.
- **Files:** `Navbar.jsx`, `Sidebar.jsx`, `App.jsx`
- **Checks & fixes:**
  - Wrap `<Sidebar>` in `<nav aria-label="Primary">`.
  - Wrap `<Navbar>` (or its inner list) in `<nav aria-label="Main navigation">`.
  - Ensure `<main>` element wraps the `<Suspense>` / `<Routes>` block (already
    partially there via `<div className="main-content">` — promote it to `<main>`).
  - Replace generic `<div class="main-content">` → `<main id="main-content" class="main-content">`.
  - Confirm `<h1>` exists on every page (at minimum a prominent page heading).
  - Avoid multiple `<h1>` — one per page preferred; page title can be the `<h1>`.
- **If needed:** inform Navbar/Sidebar owners of these changes before merging.

### Task 12 – (Optional / Phase 1.5) SSR / Prerendering audit path

- **Why:** Google now executes JS on most pages, but Bing and smaller crawlers may
  not. If the deployed frontend is behind auth-protected routes with no SSR, only
  `/login` and the public shell are crawler-visible today.
- **Options to evaluate at next planning checkpoint:**
  1. **Vite SSG / prerendering** (`vite-plugin-ssr` or `vite-plugin-ssg`)
     — generate static HTML for `/login` at minimum; good for public-facing
     marketing pages.
  2. **Next.js migration** — full SSR/SSG support, built-in `<head>` API, and
     Incremental Static Regeneration — overkill if the entire app is behind auth.
  3. **Render outside** — for public SEO content only, host a standalone marketing
     landing page alongside the SPA.
- **Decision point:** Evaluate whether any routes are ever public after auth is
  removed, or whether the app needs SEO at all outside `/login`.

## Acceptance Criteria

- [ ] `npm run build` in `Frontend/` completes with zero errors and no console
  warnings from `react-helmet-async`.
- [ ] Run production build (`npm run preview`) and inspect `<head>` on each route
  in DevTools → **every** `<title>` must be unique per page.
- [ ] Each route fires exactly **one** `<title>`, **one** `meta[name="description"]`,
  **one** `canonical`, and exactly **one** set of OG / Twitter tags (no
  doubles or orphans).
- [ ] `PageSEO` is used in **all 14 route components** — `Login` through `GapAnalysis`.
- [ ] `SITE_URL` is a single exported constant in `seoDefaults.js` used everywhere
  canonical/OG URL is needed.
- [ ] `manifest.json` is reachable at `/manifest.json` in both dev and preview builds.
- [ ] `robots.txt` and `sitemap.xml` are reachable at their canonical paths.
- [ ] All four favicon files exist and `index.html` icon references resolve with
  no 404 in the browser network panel.
- [ ] Run Lighthouse SEO audit (`npm run build && npm run preview`, Chrome
  DevTools Lighthouse → SEO category) — score must be ≥ 90/100 with zero
  criticals, zero blockers, and no "Document does not have a `<title>` element"
  or "Document does not have a meta description" warnings.
