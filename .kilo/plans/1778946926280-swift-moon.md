# API Integration Plan for Court Scheduling System

## Current State Analysis

### Frontend Structure
- React application with Context API for state management
- Custom hooks: `useQuery` (GET with caching) and `useMutation` (POST/PUT/DELETE)
- Centralized API client (`src/api/client.js`) using Axios with:
  - Base URL from environment variable `VITE_API_BASE_URL` (defaults to `http://localhost:8000`)
  - Automatic Basic Auth injection from sessionStorage
  - 401 response interceptor for session expiration
- Demo mode in Login.jsx (mock authentication)
- Inconsistent API usage: some components use direct client calls, others use hooks

### Backend Status
- No backend code visible in current repository
- API endpoints assumed to be at `http://localhost:8000` based on client configuration
- Current frontend uses mock data and simulated latency

## Required API Endpoints

Based on frontend usage, the following API endpoints are needed:

### Authentication
- `POST /login` - Authenticate user (currently mocked in Login.jsx)
- `POST /logout` - Clear session (handled client-side)
- `GET /session` - Check authentication status (could be added)

### Dashboard
- `GET /dashboard/stats` - Get dashboard statistics (used in Dashboard.jsx)

### Courtrooms
- `GET /courtrooms` - List all courtrooms
- `GET /courtrooms/:id` - Get specific courtroom
- `POST /courtrooms` - Create new courtroom
- `PUT /courtrooms/:id` - Update courtroom
- `DELETE /courtrooms/:id` - Deactivate/delete courtroom

### Cases
- `GET /cases` - List cases (with query params for filtering/pagination)
- `GET /cases/:id` - Get specific case
- `POST /cases` - Create new case
- `PUT /cases/:id` - Update case
- `DELETE /cases/:id` - Delete case
- `PATCH /cases/:id` - Partial updates (e.g., status change)
- `GET /judges` - List judges (for assignment dropdown)

### Users
- `GET /users` - List all users
- `GET /users/:id` - Get specific user
- `POST /users` - Create new user
- `PUT /users/:id` - Update user
- `DELETE /users/:id` - Delete user
- `POST /users/:id/reset-password` - Reset password (referenced in UI)

### Judges
- `GET /judges` - List all judges
- `GET /judges/:id` - Get specific judge
- `POST /judges` - Create new judge
- `PUT /judges/:id` - Update judge
- `DELETE /judges/:id` - Delete judge

### Schedule
- `GET /schedule` - Get schedule data
- `POST /schedule/generate` - Generate new schedule (referenced in Dashboard)
- `GET /schedule/:week` - Get schedule for specific week
- `POST /schedule/:week/publish` - Publish schedule

### Calendar
- `GET /calendar/events` - Get calendar events
- `POST /calendar/events` - Create calendar event
- `PUT /calendar/events/:id` - Update calendar event
- `DELETE /calendar/events/:id` - Delete calendar event

### Reports
- `GET /reports/workload` - Get judge workload report
- `GET /reports/case-stats` - Get case statistics
- `GET /reports/utilization` - Get courtroom utilization
- `EXPORT /reports/:format` - Export reports (PDF/CSV)

### AI Insights (from AIInsights.jsx)
- `GET /ai/insights` - Get AI-generated insights
- `POST /ai/predict` - Get case duration prediction
- `POST /ai/suggest` - Get scheduling suggestions

### Settings
- `GET /settings` - Get application settings
- `PUT /settings` - Update application settings
- `GET /preferences` - Get user preferences
- `PUT /preferences` - Update user preferences

3. [x] Implement proper error handling and loading states in hooks

## Technical Recommendations

### Hook Enhancements
1. Extend `useQuery` to support:
   - Pagination parameters
   - Query parameter merging
   - Cache invalidation strategies
   - Stale-while-revalidate caching

2. Extend `useMutation` to support:
   - Optimistic updates with rollback
   - Transactional mutations
   - Polling for long-running operations

### API Client Improvements
1. Add request timeout configuration
2. Implement exponential backoff for retries
3. Add request queuing for offline support
4. Implement request batching where beneficial

### Data Structure Standards
1. Define consistent response formats:
   ```json
   {
     "success": true,
     "data": {/* resource data */},
     "message": "Optional message",
     "timestamp": "ISO timestamp"
   }
   ```
2. Standardize error responses:
   ```json
   {
     "success": false,
     "error": {
       "code": "ERROR_CODE",
       "message": "Human readable message",
       "details": {/* optional validation details */}
     }
   }
   ```
3. Use proper HTTP status codes
4. Implement pagination with cursor or offset/limit

## Security Considerations
1. Implement proper CORS policies
2. Add rate limiting on API endpoints
3. Validate and sanitize all inputs
4. Use HTTPS in production
5. Implement proper authentication (JWT or session-based)
6. Add authorization middleware for role-based access
